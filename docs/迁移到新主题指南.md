# 迁移到新主题指南

## 定位

本指南说明如何把 TARA 从当前惩罚性赔偿 topic instance 迁移到新的法律或合规研究主题。迁移不是复制旧主题结论，而是复用研究对象模型、组装动作和质量门禁。

## 什么时候适合迁移 TARA

适合：

- 材料来源多、权威层级复杂。
- 需要区分事实、规则、观点和结论。
- 需要把 AI 输出控制在待复核草稿范围内。
- 需要形成判断包、风险包或数据缺口清单。
- 需要未来跨主题复用工作流。

不适合：

- 只有单一材料摘要任务。
- 用户只需要最终展示排版。
- 没有原始来源或无法保留来源路径。
- 研究目标要求 AI 直接作最终法律判断。

## 新主题启动前的 topic profile

新主题不得直接复制旧主题的观察因素、案例结论或 pre 范围。启动前先建立 topic profile，至少包含：

- 研究目的：
- 目标读者：
- 工作模式：
- 来源层级：
- 材料新鲜度要求：
- 允许表述：
- 禁止表述：
- 人工复核触发点：
- 最小试跑范围：

topic profile 形成后，再进入 topic config、source intake、对象提取和 pilot。

## Source hierarchy

每个新主题都必须先定义来源层级。推荐顺序：

1. 官方规则、法律法规、监管文件、司法解释、国际组织官方文件。
2. 权威案例、指导案例、案例库参考案例。
3. 普通裁判文书或行政决定。
4. 公司年报、公告、审计报告、披露文件。
5. 学术论文、机构报告。
6. 律所文章、媒体报道、公众号、二手整理。

低权威材料可以作为线索、背景或问题清单，但不能单独支撑最终法律结论。

## 工作模式一：case-factor mode

适用场景：

- 司法案例研究
- 裁判规则观察
- 案例横向比较
- 课程 pre
- 判决分析

链条：

```text
case
→ fact
→ evidence
→ factor
→ decision packet
→ comparison
→ quality review
```

典型对象：

- 来源卡
- 事实卡
- 证据卡
- 裁判因素卡
- 案例因素包
- 判断包
- 风险包

## 工作模式二：compliance-mapping mode

适用场景：

- Temu/PDD 双支柱
- 数据合规
- 数据出境
- 平台合规
- 游戏合规
- 跨境电商合规
- 出海合规

链条：

```text
official rule / company source / company fact
→ source hierarchy
→ rule atoms
→ fact atoms
→ issue map
→ rule-fact bindings
→ data gap checklist
→ decision packet
→ quality review
```

典型对象：

- 来源卡
- 规则卡
- 事实卡
- 争点卡
- 证据卡
- 判断包
- 风险包
- 数据缺口清单

## 如何从旧主题迁移

可以复用：

- 五层目录边界。
- 来源卡、事实卡、规则卡、争点卡、证据卡、判断包、风险包。
- allowed claim / forbidden claim 机制。
- uncertainty / data_gap / next_action / human_review_status 字段。
- quality review 维度。

不得复用：

- 旧主题的具体裁判因素。
- 旧主题的案例结论。
- 旧主题的赔偿表达方式。
- 旧主题的 pre 主线。
- 旧主题中未核验的 rule anchor。

## 什么时候必须停止

出现以下情况时，不得进入判断包升级或 final synthesis：

- 缺原始来源。
- 来源权威不足以支撑表述强度。
- 出现高强度结论但无证据。
- 法规、政策或公司披露可能过期。
- AI 输出看起来像最终法律意见。
- rule anchor 仍是 pending_backfill。
- 判断包缺少 allowed claim、forbidden claim、data gap 或 human review status。

处理方式：

- 标记“待核验”。
- 降低表述强度。
- 启动 backfill。
- 补充 topic profile。
- 等待人工复核。

## 新主题最小试跑

新主题最小试跑不应全量迁移。建议：

1. 选 1-2 个代表性材料包。
2. 建立 source atoms。
3. 建立 rule atoms 或 factor atoms。
4. 建立 fact atoms 和 evidence atoms。
5. 生成 2-3 个 decision packets。
6. 生成 quality review。
7. 写 assembly notes，说明哪些内容可进入后续输出，哪些必须停止。

只有最小试跑通过后，才进入批量处理。
